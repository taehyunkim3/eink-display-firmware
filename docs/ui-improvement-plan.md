# E-ink 대시보드 UI/폰트/성능 개선 계획

작성일: 2026-07-04
대상: `eink-firmware` (네이티브 렌더링), 일부 `eink-frontend` (데이터 공급)

---

## 1. 현황 진단

### 1-1. 폰트가 깨져 보이는 문제 ("택" → "틱")

**근본 원인: Galmuri TTF를 잘못된 크기로 래스터라이즈하고 있음.**

Galmuri 공식 문서(https://github.com/quiple/galmuri) 기준, 각 폰트가 픽셀 퍼펙트로
나오는 크기는 아래와 같다.

| 폰트 | 픽셀 퍼펙트 크기 | 현재 생성 크기 (`tools/fonts/generate_bitmap_font.py`) |
|---|---|---|
| Galmuri14 | **15px** (11pt) | 14px ← 잘못됨 |
| Galmuri11 / 11 Bold | **12px** (9pt) | 11px ← 잘못됨 |
| Galmuri9 | **10px** (7.5pt) | 9px ← 잘못됨 |
| Galmuri7 | **8px** (6pt) | 7px ← 잘못됨 |

전부 1px씩 작게 렌더링하고 있어서 비정수 배율 축소가 일어나고, 안티앨리어싱된
회색 픽셀을 `threshold=96`으로 이진화하면서 획이 통째로 사라지거나 붙는다.
"택"의 ㅐ 가로획이 소실되어 "틱"처럼 보이는 것이 정확히 이 증상이다.

**부차 원인 (같은 스크립트, `glyph_bytes()`):**

- 글리프마다 `textbbox` 기준 개별 중앙정렬을 해서 **베이스라인이 글자마다 흔들린다**
  (p, g 같은 디센더나 받침 유무에 따라 글자가 위아래로 튐).
- ASCII도 한글과 같은 정사각 셀 폭(`glyphSize`)으로 전진해서 **영문/숫자 자간이
  지나치게 벌어진다** (숫자 "1"도 14px 폭 차지). `spaceWidth`만 별도 처리 중.

**개선 여지:** Galmuri는 릴리스에 **BDF(비트맵 원본)** 파일을 제공한다. TTF를
다시 래스터라이즈할 필요 없이 BDF에서 비트맵을 그대로 추출하면 100% 원본과
동일한 픽셀이 나오고, 글리프별 실제 폭(proportional advance)도 얻을 수 있다.

### 1-2. 주식 차트: 가격이 x축에 있음

`src/main.cpp`의 `drawChartPointLabels()`가 "시간 가격" 쌍 3개(시작/중간/끝)를
차트 **아래쪽(x축 위치)** 에 출력한다. y축 가격 눈금은 존재하지 않는다.
차트는 전일 종가 대비 % 스케일(`percentY`)로 그려지므로, y축에는
%든 가격이든 눈금을 붙일 수 있는 구조는 이미 갖춰져 있다.

### 1-3. 작은 숫자 폰트

`TextSize::Micro`(Galmuri7, 7px 셀)가 **이미 정의되어 있으나 사용처가 0곳**이다.
축 라벨용으로 준비만 되어 있는 상태. 단, 1-1과 같은 이유로 현재 Micro 비트맵도
깨져 있을 가능성이 높으므로 폰트 재생성이 선행되어야 한다.

### 1-4. 느린 이유

버튼으로 페이지를 넘길 때마다 `refreshScreen()`이 전 과정을 반복한다:

1. Wi-Fi 연결 확인 (`connectWifi`)
2. 서버에서 대시보드 JSON **전체를 매번 재다운로드** (`fetchJson`) — 로컬 캐시 없음
3. JSON 파싱 (`deserializeJson`)
4. 렌더링 + e-ink 갱신 (풀 리프레시 ~2초는 하드웨어 고유 시간)

페이지 전환은 같은 데이터를 다른 레이아웃으로 그리는 것뿐인데 네트워크 왕복이
끼어 있어 체감 지연이 크다. 부가적으로 글리프 렌더링이 픽셀 단위
`display.drawPixel()` 호출이라 CPU 시간도 낭비된다 (수천 자 × 픽셀 수).

### 1-5. 날씨 아이콘

`writeDefaultWeatherIcon()`이 코드로 원/선을 조합해 32×32 PBM 3종(해/구름/비)만
생성한다. 문제점:

- 해상도가 32px로 작고 선이 1px라 e-ink에서 빈약해 보임
- 날씨 구분이 3종뿐 (눈, 안개, 뇌우, 부분 흐림 등이 모두 "구름"으로 표시)
- SD 카드에 한 번 쓰면 갱신되지 않음 (`SD.exists` 체크로 스킵)

### 1-6. 전반적인 UI 인상 ("투박하고 휑함")

- 모든 페이지가 1px 직선 프레임 + 텍스트 나열 구조. 시각적 위계(굵기 대비,
  반전 블록, 큰 숫자 강조)가 헤더 타이틀 한 곳뿐.
- 숫자 정보(온도, 주가)가 본문 텍스트와 같은 크기라 눈에 안 들어옴.
- 여백/정렬 그리드가 페이지마다 제각각 (x=24, 28, 32, 36 혼재).
- 800×480 해상도 대비 정보 밀도가 낮은 페이지(요약)와 과밀한 페이지(캘린더)가 공존.

## 2. 개선 계획

### Phase 1 — 폰트 품질 (최우선, 다른 작업의 전제)

| # | 작업 | 상세 |
|---|---|---|
| 1-1 | 폰트 생성 스크립트를 BDF 기반으로 전환 | `tools/fonts/generate_bitmap_font.py`가 Galmuri 릴리스의 `.bdf`를 파싱해 원본 비트맵을 그대로 추출. 렌더링/threshold 제거 → 픽셀 왜곡 원천 차단 |
| 1-2 | 글리프별 실제 폭(advance) 저장 | 헤더에 글리프당 폭 테이블(uint8 배열) 추가. ASCII proportional 전진으로 자간 정상화. 한글은 고정폭 유지 |
| 1-3 | 베이스라인 통일 | BDF의 `BBX`/`DWIDTH` 오프셋을 사용해 모든 글리프가 같은 베이스라인에 정렬되도록 생성 |
| 1-4 | 4종 크기 재생성 및 교체 | Galmuri7(8px)/9(10px)/11(12px)/11Bold(12px)/14(15px) → `include/generated/*.h` 재생성. `bitmapFontForSize()`의 `glyphSize`/`spaceWidth` 상수 보정 |
| 1-5 | (선택) 대체 폰트 비교 샘플 | Neo둥근모(16px, 옛 PC통신 감성·가독성 높음), 둥근모+Fixedsys 후보. 같은 파이프라인으로 헤더를 만들어 실기기에서 비교 후 결정 |

폰트 후보 비교 (https://geminian.tistory.com/m/33 기준):

| 후보 | 장점 | 단점 |
|---|---|---|
| **Galmuri (현행, 크기만 수정)** | 5개 크기 세트 완비, 이미 파이프라인 존재 | DS풍이라 다소 게임스러움 |
| **Neo둥근모** | 가독성·완성도 최고, 전광판에도 쓰이는 검증된 폰트, OFL | 16px 단일 크기 (작은 라벨은 Galmuri7/9 병용 필요) |
| PF스타더스트 | 얇고 깔끔 | 크기 옵션 부족, 비트맵 원본 없음 |
| LanaPixel | 9px 컴팩트 | 한글 조형이 Galmuri9와 대동소이 |

권장: **본문/제목은 Neo둥근모(16px), 라벨/축 숫자는 Galmuri9(10px)/Galmuri7(8px)**
조합이 가독성과 정보 밀도 균형이 가장 좋음. 다만 폰트 세트가 둘로 늘어나는
부담이 있으므로, 1-4까지 끝낸 "제대로 나오는 Galmuri"를 먼저 보고 결정해도 됨.

### Phase 2 — 주식 차트 축 정리

| # | 작업 | 상세 |
|---|---|---|
| 2-1 | y축 가격 눈금 추가 | 차트 우측(또는 좌측)에 상/중(전일종가)/하 3단 가격 라벨. `percentY` 역산으로 가격 계산. `TextSize::Micro` 사용 |
| 2-2 | x축은 시간만 표시 | `drawChartPointLabels()`에서 가격 제거, 시작/중간/끝 시각만 남김. Micro 폰트 적용 |
| 2-3 | 차트 폭 보정 | y축 라벨 영역(약 40px)만큼 차트 본체 폭 축소 |
| 2-4 | 프론트엔드(`DetailedStockChart`)도 동일 규칙 적용 | 웹 미리보기와 기기 화면의 축 표기 일치 |

### Phase 3 — 체감 속도

| # | 작업 | 상세 |
|---|---|---|
| 3-1 | JSON 로컬 캐시 | 마지막 성공 JSON을 RAM(또는 RTC/heap)에 보관. **페이지 전환 시 네트워크 생략**하고 즉시 렌더. 데이터는 주기 갱신(refreshSeconds) 또는 새로고침 버튼에서만 재요청 |
| 3-2 | 페이지 전환은 항상 partial refresh | 현재도 조건부 지원(`shouldUsePartialRefreshForScreenTransition`) — 캐시와 결합하면 전환 체감이 "버튼 → 약 1초"로 단축 |
| 3-3 | 글리프 렌더링 최적화 | 픽셀 단위 `drawPixel` → 행 단위 비트 연산 또는 `drawBitmap` 사용. 텍스트 많은 페이지(캘린더)에서 렌더 시간 단축 |
| 3-4 | (측정 후 판단) fetch 병렬화·HTTP keep-alive | 3-1 적용 후에도 주기 갱신이 느리면 검토 |

### Phase 4 — 날씨 아이콘

| # | 작업 | 상세 |
|---|---|---|
| 4-1 | 아이콘 소스를 사전 제작 비트맵으로 교체 | 코드 생성 대신 오픈소스 픽셀 날씨 아이콘 세트(예: Meteocons 픽셀 변환, 또는 직접 도트 제작)를 PBM으로 변환해 펌웨어에 내장(PROGMEM). SD 의존 제거 |
| 4-2 | 날씨 코드 매핑 세분화 | WMO 코드 기준 최소 7종: 맑음/부분흐림/흐림/비/소나기/눈/뇌우/안개 |
| 4-3 | 크기 2종 | 요약 페이지용 48px, 주간 날씨 셀용 32px |
| 4-4 | 기존 SD 아이콘 갱신 문제 해결 | 버전 파일명 대신 펌웨어 내장으로 전환하면 자동 해결 |

### Phase 5 — 레이아웃/위계 다듬기

| # | 작업 | 상세 |
|---|---|---|
| 5-1 | 공통 그리드 상수 정의 | 페이지 여백(예: 16px), 셀 패딩, 행 높이를 상수화해 전 페이지 통일 |
| 5-2 | 숫자 강조 | 요약 페이지 온도·주가 등 핵심 숫자를 Large(15px)+Bold 조합으로 크게. 라벨은 Tiny로 대비 |
| 5-3 | 섹션 헤더 스타일 통일 | 반전 블록(현 헤더 타이틀 방식)을 섹션 제목에도 일관 적용 |
| 5-4 | 요약 페이지 밀도 개선 | 좌: 날씨(큰 아이콘+큰 온도), 우: 시장 요약, 하: 일정 3건 — 빈 공간 축소 |
| 5-5 | 구분선 다이어트 | 셀마다 4변 프레임 대신 수평 구분선 위주로 정리해 답답함 완화 |

## 3. 우선순위와 순서

```
Phase 1 (폰트)  ──► Phase 2 (차트 축)  ──► Phase 5 (레이아웃)
      │
      └──► Phase 3 (속도) ── 폰트와 독립적, 병행 가능
Phase 4 (아이콘) ── 독립적, 아무 때나 가능
```

1. **Phase 1**: 모든 화면 품질의 전제. 이것만으로 "이글거림 없이 선명 + 조형 정상"이 달성됨.
2. **Phase 3-1, 3-2**: 코드 변경량 대비 체감 효과가 가장 큼 (페이지 전환 즉시화).
3. **Phase 2**: 사용자 지적 사항 직접 해결.
4. **Phase 4, 5**: 인상 개선.

## 4. 검증 방법

- 폰트: "택시 틱택톡 뷁 1234567890 ABCdef" 테스트 문자열을 4크기로 찍는
  디버그 페이지(또는 부팅 테스트 화면 활용)로 실기기 확인.
- 속도: 페이지 전환 시간을 `millis()` 로그로 전/후 비교 (목표: 네트워크 0ms).
- 차트: 프론트 `screen.json` 실데이터로 y축 가격 3단 라벨 위치 확인.

## 5. 확정된 사항 (2026-07-04)

1. 폰트: Galmuri BDF 기반 재생성으로 진행 (Neo둥근모는 결과 확인 후 재검토).
2. y축 가격 라벨: 우측.
3. 캐시 정책: 사용하되 30분 TTL 초과 시 자동 재요청.
